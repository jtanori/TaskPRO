# CAP-001 — Capability Map & Screen-to-Module Matrix v0.1

## Purpose

This document becomes the **master blueprint** of the application.

It connects every architectural layer we've defined so far:

* Domain
* Database
* APIs
* Modules
* Navigation
* Screens
* Permissions
* Events
* Development roadmap

Nothing should be implemented unless it belongs to a capability.

Think of this as the "table of contents" for the entire product.

---

# Capability Hierarchy

TaskPRO should be organized into four levels.

```text
Platform

    ↓

Capability

    ↓

Feature

    ↓

Screen
```

Example

```text
Booking

    ↓

Booking Lifecycle

    ↓

Accept Booking

    ↓

Booking Details Screen
```

This prevents "screen-driven development."

# Capability Dependencies

Rather than allowing arbitrary dependencies, define them explicitly.

```text
Authentication
        │
        ▼
User Profile
        │
        ▼
Marketplace
        │
        ▼
Professional Discovery
        │
        ▼
Booking
        │
        ├──────────────┐
        ▼              ▼
Messaging        Payments
        │              │
        └──────┬───────┘
               ▼
           Reviews
```

Notice:

* Messaging does **not** depend on Payments.
* Reviews depend on completed bookings.
* Booking is the central capability.

This gives objective readiness indicators for every capability.

# Capability Registry

I recommend maintaining a single machine-readable registry (YAML or JSON) describing every capability.

Example:

```yaml
booking:
  module: booking
  aggregate: Booking
  routes:
    - booking/list
    - booking/details
    - booking/history
  permissions:
    - booking:create
    - booking:update
  events:
    - BookingRequested
    - BookingCompleted
```

From this registry we can automatically generate:

* Expo Router configuration
* Navigation menus
* Permission mappings
* API documentation
* Test scaffolds
* Dependency graphs
* Implementation checklists

This turns the capability map into an executable architectural artifact rather than a static document.

# Architectural Observation

At this point, we've effectively defined the **Product Architecture**. The next major step should not be writing React Native code—it should be defining the **Design System Architecture** that will sit between these capabilities and the UI.

Specifically, I recommend creating a dedicated **UI Architecture series** that maps your existing designs into reusable implementation primitives instead of individual screens.

That series would include:

* **UI-001** — Application Shell & Navigation Architecture
* **UI-002** — Design Token Specification
* **UI-003** — Component Library Architecture
* **UI-004** — Screen Composition Patterns
* **UI-005** — Navigation Graph & User Flows
* **UI-006** — Screen Development Roadmap (derived directly from the designs)

This is where your existing design work and the architecture we've defined finally converge into an implementation-ready system.
