# UI-003 — Component Library Architecture & Primitive Catalog v0.1

## Purpose

This document defines the **implementation architecture** of the TaskPRO Design System.

It is **not** a component inventory.

It defines:

* Component hierarchy
* Ownership
* Composition rules
* API conventions
* Variant system
* Extensibility

The objective is to create a library that can build **100% of the application** without requiring one-off UI components.

---

# Design Philosophy

The component library follows a simple rule:

> **Composition over customization.**

Instead of building:

* BookingButton
* PaymentButton
* AIButton
* PremiumButton

We build:

```text
Button

↓

Variants

↓

Composition
```

The design system owns presentation.

Modules own business meaning.

---

# Component Hierarchy

The library has six layers.

```text
Design Tokens

↓

Primitives

↓

Foundation Components

↓

Composite Components

↓

Patterns

↓

```

Only the first four belong to the Design System.

# Layer 1 — Primitives

These wrap React Native.

Nothing more.


Never React Native directly.

Example:



# Missing Layer: Layout Primitives

After reviewing both the architecture and the UI direction, I believe one important layer is still missing from most modern design systems: **Layout Primitives**.

Between **Primitives** and **Foundation Components**, I would introduce reusable layout objects.

Examples:

```text
Screen

Section

Container

Panel

Sidebar

Canvas

Inspector

Toolbar

Floating Dock

Command Palette

Split View

Resizable Panel
```

These are not business components and they are not simple primitives—they are architectural building blocks. They define how entire screens are assembled.

For TaskPRO, this is especially important because many screens (dashboard, booking workspace, AI assistant, profile editor) reuse the same structural layouts. Having these as first-class components dramatically reduces duplication and ensures every screen follows a consistent composition model.

This addition also aligns perfectly with the visual direction you've been developing in Gemini, where the workspace itself is a reusable design primitive rather than something recreated for every screen.
