# TASKPRO-STATE-001 — TaskPRO Application State & Workflow State Machine Specification v0.1

## Purpose

The State Specification defines how TaskPRO represents, transitions, synchronizes, and persists application state.

This artifact connects:

```text id="m2u7pk"
TASKPRO-FLOW-001
Workflow Specification

        ↓

TASKPRO-DATA-001
Domain Model

        ↓

TASKPRO-STATE-001
State Architecture

        ↓

TASKPRO-ARCH-001
Application Runtime
```

The objective is to prevent inconsistent UI behavior by defining:

* state ownership
* state transitions
* lifecycle rules
* synchronization behavior
* offline handling
* event propagation

---

# 1. State Architecture Overview

## 1.1 State Principles

TaskPRO state follows five principles:

---

## Explicit State

Every meaningful workflow state is represented.

Avoid:

```text id="9m7r83"
boolean flags everywhere
```

Prefer:

```text id="b9t7qk"
BookingStatus = Scheduled
```

---

## Single Ownership

Each state has a clear owner.

Example:

```text id="7c0i6p"
Booking status

Owned by:
Booking domain

Consumed by:
Screens
Notifications
Tasks
```

---

## Predictable Transitions

States move through defined paths.

Invalid transitions are rejected.

---

## Observable State

Important state changes generate events.

Example:

```text id="z3wq6e"
Booking Accepted

↓

BOOKING_ACCEPTED event

↓

Notification

↓

UI update
```

---

## Recoverable State

Users should recover from:

* app restart
* network loss
* interrupted workflows

---

Examples:


Mobile stores:

* cached copy
* synchronization metadata

# 6. Workflow State

## Purpose

Represents business process progress.

Workflow state is the most important state category.

---

# 6.1 Booking State Machine

## Entity

Booking

---

## States

```text id="0f7w5n"

↓

SUBMITTED

↓

ACCEPTED

↓

SCHEDULED

↓

IN_PROGRESS

↓

```

Alternative termination:

```text id="w7r3nb"

```

---

# Transition Rules

## Draft → Submitted

Requirement:

* service selected
* required information complete

Event:

```text id="r8h3p9"
BOOKING_CREATED
```

---

## Submitted → Accepted

Actor:

Provider

Event:

```text id="h8k4ws"
BOOKING_ACCEPTED
```

---

## Accepted → Scheduled

Requirement:

* date/time confirmed

Event:

```text id="x0j3ns"
BOOKING_SCHEDULED
```

---

## Scheduled → In Progress

Actor:

Provider

Event:

```text id="r3n7jd"
TASK_STARTED
```

---

## In Progress → Completed

Requirement:

* work finished

Event:

```text id="5j8q0a"
BOOKING_COMPLETED
```

---

# 10. State Persistence Rules

## Persist

Long-lived state:

* authentication
* preferences
* pending workflows
* cached entities

---

## Do Not Persist

Temporary state:

* modal visibility
* animations
* temporary selections
