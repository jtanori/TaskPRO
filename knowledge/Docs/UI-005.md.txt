# UI-005 — Navigation Graph, User Journeys & Interaction Architecture v0.1

## Purpose

This document defines **how users move through TaskPRO**.

Unlike UI-001 (Application Shell), this document defines:

* Navigation Graph
* User Journeys
* Capability transitions
* Interaction rules
* Deep linking
* Flow orchestration

It is the behavioral blueprint of the application.

---

# Design Philosophy

Navigation should never feel like moving between pages.

Instead, users move between **contexts** while maintaining continuity.

The navigation model should optimize for:

* Task completion
* Context preservation
* AI assistance
* Progressive disclosure
* Minimal cognitive load

---

# Navigation Rules

## Rule 1

Never navigate after a successful operation unless the workflow changes.

Example:

After updating profile:

❌ Navigate away

✅ Stay in context and show confirmation.

---

## Rule 2

Business events drive navigation.

Example:

```text
BookingCompleted

↓

Invoice Screen
```

Not:

```text
Button Press

↓

Invoice
```

---

## Rule 3

Users should never lose work.

Forms persist automatically.

---

## Rule 4

Every long-running operation is resumable.

Examples:

Booking

Registration

Evidence Upload

Payment

---

# Modal Navigation

Use overlays only for transient interactions.

Allowed:

```text
Confirmation

Date Picker

Filters

Photo Viewer

Scanner

Share
```

Not allowed:

```text
Entire Booking Flow

Entire Profile

Checkout
```

Complex workflows deserve their own navigation context.

---

# Deep Linking

Every capability exposes canonical routes.

Examples:

```text
taskpro://booking/{id}

taskpro://professional/{id}

taskpro://conversation/{id}

taskpro://invoice/{id}

taskpro://review/{id}
```

Deep links always restore the correct context.

---

# Universal Search Navigation

Search is capability-agnostic.

Example:

```text
Search

↓

Booking

↓

Booking Context
```

or

```text
Search

↓

Professional

↓

Professional Context
```

Search never becomes a dead end.

# AI Navigation

Examples:

```text

↓
```

```text

↓
```

# Interaction Architecture

Every user interaction belongs to one of five categories.

```text
Navigate

Command

Selection

Manipulation

Conversation
```

Examples:

| Interaction         | Category     |
| ------------------- | ------------ |
| Open Booking        | Navigate     |
| Cancel Booking      | Command      |
| Select Professional | Selection    |
| Upload Photo        | Manipulation |
| Ask AI              | Conversation |

This classification simplifies gesture design and accessibility.

# Flow Recovery

If the app is interrupted:

```text
App Closed

↓

Restore Session

↓

Restore Context

↓

Restore Flow

↓

Restore Drafts
```

A user returning after several hours should continue exactly where they left off whenever possible.

# Navigation Registry

Maintain a declarative registry for routes.

Example:

```yaml
booking.details:
  capability: booking
  type: detail
  permissions:
    - booking:view
  deepLink: taskpro://booking/{id}
  supportsAI: true
  resumable: true
```

This registry becomes the source for:

* Expo Router configuration
* Deep link configuration
* Permission guards
* Analytics
* Documentation

# Architectural Observation

At this point, the application architecture is no longer screen-centric—it is **flow-centric**.

The final document in this UI architecture series should capitalize on that by producing a **screen implementation roadmap** derived from the capability map, navigation graph, and your existing designs.

Rather than a flat backlog of screens, it should define:

* implementation order,
* dependencies,
* reusable archetypes,
* shared components,
* estimated effort,
* and parallel development opportunities.

This transforms the UI from a collection of mockups into an executable engineering plan.
