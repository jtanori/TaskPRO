# STATE-001 — Client State Management & Offline Synchronization Architecture v0.1

## Purpose

The client architecture should treat the mobile application as a **local-first application** rather than a thin API client.

The user should never feel like they are interacting with a remote database.

Instead, the application behaves like this:

```text
User

↓

Local State

↓

Optimistic Update

↓

Background Synchronization

↓

Backend Confirmation

↓

Realtime Updates
```

The UI should respond immediately while the synchronization layer reconciles changes.

---

# Design Principles

The state architecture should satisfy these goals:

* Local-first
* Offline capable
* Optimistic by default
* Event driven
* Predictable
* Recoverable
* Replayable
* Testable

The UI never owns business state.

---

# Layered State Architecture

```text
UI Components

↓

Presentation State

↓

Application State

↓

Domain Cache

↓

Synchronization Engine

↓

Supabase
```

Each layer has a single responsibility.

---

# State Categories

Instead of a single global store, TaskPRO should separate state into five categories.

```text
UI State

Session State

Domain State

Cache State

Synchronization State
```

Each has a different lifecycle.

---

# 1. UI State

Ephemeral state that exists only while a screen is active.

Examples:

```text
Modal Open

Current Tab

Selected Card

Expanded Panel

Search Text

Bottom Sheet Position

Animation State
```

Technology:

* React State
* React Context (when shared within a feature)

Never persist UI state.

---

# 2. Session State

Lives while the user is authenticated.

Contains:

```text
Current User

Permissions

Feature Flags

Current Organization

Current Professional Context

Theme

Locale
```

Technology:

* Zustand

Persisted:

* Yes

Storage:

* MMKV + SecureStore (for sensitive values)

---

# 3. Domain State

Represents business entities.

Examples:

```text
Bookings

Professionals

Messages

Reviews

Payments

Notifications

Services
```

Technology:

* TanStack Query
* Normalized entity cache
* Repository pattern

Never manually duplicate server state in multiple stores.

---

# 4. Cache State

Stores projections optimized for reading.

Examples:

```text
Recent Searches

Nearby Professionals

Service Categories

Home Dashboard

Recommendations
```

Technology:

* TanStack Query
* Persistent cache
* Background refresh

---

# 5. Synchronization State

Responsible for:

```text
Pending Commands

Retry Queue

Conflict Queue

Upload Queue

Download Queue

Sync Status
```

This layer should be completely invisible to the UI.

# Offline Queue

Every write operation becomes a command.

Example:

```text

UploadEvidenceCommand

SendMessageCommand

CompleteBookingCommand
```

Commands are stored locally until synchronization succeeds.

# Architectural Observation

Rather than implementing offline support as an afterthought, I recommend making **commands** the fundamental unit of client-side state changes.

Every user action—booking a service, uploading evidence, sending a message, updating availability—should first become a **Command** stored locally. The synchronization engine is then responsible for delivering those commands to the backend and reconciling the results.

This mirrors the event-driven architecture established in `DOM-001` and `API-001`, provides robust offline behavior, and significantly simplifies debugging because every state transition originates from a recorded user intent.
