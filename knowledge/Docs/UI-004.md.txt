# UI-004 — Screen Composition System & Layout Primitives v0.1

## Purpose

This document defines **how every screen in TaskPRO is constructed**.

It sits between:

* **UI-003** (Component Library)
* **Individual Screen Specifications**

Instead of designing screens independently, every screen becomes a composition of reusable **Layout Primitives** and **Composition Patterns**.

This document is arguably one of the most important in the entire UI architecture because it makes the UI **systematic instead of artistic**.

---

# Philosophy

The application should never ask:

> "How do we design this screen?"

Instead, it asks:

> "Which composition pattern does this screen use?"

Every screen becomes a predictable arrangement of reusable building blocks.

---

# Composition Hierarchy

Every screen is assembled from six layers.

```text
Design Tokens

↓

Primitives

↓

Foundation Components

↓

Layout Primitives

↓

Composition Patterns

↓

```

Notice that **Layout Primitives** now become a first-class architectural layer.

# The 12 Layout Primitives

After analyzing the TaskPRO designs, I believe almost every screen can be composed from only twelve structural primitives.

# Screen Metadata

Every screen should declare metadata.

Example

```yaml
screen: Booking Details

type: Detail

supportsAI: true

supportsOffline: true

supportsInspector: true

supportsDock: true

permissions:
  - booking:view

capability:
  - booking
```

This metadata can drive routing, permissions, analytics, and even AI-assisted testing.

---

# Screen Composition DSL

Rather than manually building every screen, I recommend introducing a declarative composition model.

Example:

```yaml
screen:
  layout: detail

header:
  variant: large

sections:
  - booking-summary
  - timeline
  - payment
  - evidence

dock:
  - camera
  - assistant

inspector:
  booking-context

overlays:
  - image-viewer
```

This opens the door to:

* screen generation
* automated documentation
* design validation
* visual regression
* AI-assisted scaffolding

---

# Architectural Observation

This is the point where I would make one significant architectural leap beyond a traditional design system.

Instead of documenting **screens**, we should document **screen archetypes** and **screen blueprints**.

That means the next layer is no longer:

> "Build the Booking Screen."

Instead it becomes:

> **"Instantiate the Detail Archetype using the Booking capability."**

This separation means:

* UI evolves independently from business logic.
* New capabilities automatically inherit proven layouts.
* AI can generate production-ready screens from metadata rather than handcrafted prompts.
* Design, engineering, and documentation all speak the same language.

This approach aligns perfectly with your original vision of using Gemini to generate foundational visual primitives first and then composing complete interfaces from those primitives.
