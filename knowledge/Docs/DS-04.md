# TASKPRO-DS-004 — TaskPRO Motion Guidelines v1.0

**Status:** Approved Architecture
**Owner:** Design System
**Audience:** Product Design, Motion Design, React Native, QA, AI Code Generation

---

# 1. Purpose

Motion is a functional component of the TaskPRO user experience. Animations communicate changes in state, reinforce spatial relationships, guide user attention, and provide immediate feedback without distracting from task completion.

Motion must always improve usability. Decorative animation that does not communicate meaning should be avoided.

All animations must be deterministic, performant, accessible, and derived from the design token system defined in **TASKPRO-DS-001**.

---

# 2. Motion Principles

TaskPRO motion follows six core principles.

## 2.1 Purposeful

Every animation communicates:

* State changes
* Navigation
* Progress
* Hierarchy
* Feedback
* Continuity

If motion does not improve understanding, it should not exist.

---

## 2.2 Fast

The application should feel responsive.

Animation should never delay interaction.

Users should always feel in control.

---

## 2.3 Natural

Movement should resemble physical behavior.

Examples:

* spring settling
* inertia
* easing
* momentum

Avoid robotic linear animations.

---

## 2.4 Consistent

The same transition always behaves the same way.

Example:

Every modal enters identically.

Every bottom sheet dismisses identically.

---

## 2.5 Subtle

Motion supports the interface.

It never becomes the focus.

---

## 2.6 Accessible

Every animation must support Reduced Motion.

No animation may be required to understand application state.

# 4. Duration Tokens

## Instant

```text
75 ms
```

Used for:

* Icon changes
* Selection states

---

## Fast

```text
150 ms
```

Used for:

* Buttons
* Chips
* Press states

---

## Standard

```text
250 ms
```

Used for:

* Cards
* Lists
* Navigation

---

## Slow

```text
400 ms
```

Used for:

* Modals
* Large transitions

---

## Extended

```text
600 ms
```

Used only for:

* Onboarding
* Hero transitions
* Educational flows

---

# 5. Easing Curves

## Standard

Used for most transitions.

```text
Ease In Out
```

---

## Enter

```text
Ease Out
```

Objects decelerate into place.

---

## Exit

```text
Ease In
```

Objects accelerate away.

---

## Spring

Used for:

* Bottom sheets
* Pull-to-refresh
* Drag interactions
* Cards

Must use platform-native spring physics where possible.

---

# 6. Navigation Transitions

Navigation preserves spatial continuity.

Push navigation

```text
Right

↓

Center
```

Pop navigation

```text
Center

↓

Right
```

Transitions should complete within:

```text
250 ms
```

---

# 7. Tab Navigation

Switching tabs should never feel like launching a new application.

Rules:

* Preserve scroll position
* Preserve navigation stack
* Crossfade content
* Highlight active tab immediately

Maximum transition:

```text
150 ms
```

---

# 8. Bottom Sheet Motion

Bottom sheets use spring animation.

Presentation

```text
Bottom

↓

Snap Point
```

Dismiss

```text
Snap

↓

Bottom
```

Supports:

* drag velocity
* momentum
* overscroll resistance

---

# 9. Modal Presentation

Modals fade while slightly scaling.

Presentation

```text
Opacity

0 → 100%

Scale

96% → 100%
```

Dismissal reverses the transition.

Background scrim fades simultaneously.

---

# 10. Dialog Motion

Dialogs should feel lightweight.

Fade

*

Scale

Duration

```text
200 ms
```

# 11. Shared Element Transitions

Used sparingly.

Recommended for:

* Service cards
* Technician profile
* Booking confirmation
* Images

Not recommended for:

* Forms
* Settings
* Administrative screens

Shared elements preserve:

* position
* size
* elevation
* clipping

---

# 12. Card Expansion

Cards expand from their original position.

Animation sequence:

```text

↓

Scale

↓

Content Reveal
```

Content should never "pop" into existence.

# 23. AI Motion

AI interactions should feel conversational rather than mechanical.

Examples:

Thinking indicator

Animated pulse

Streaming response

Progressive text reveal

Suggestion appearance

Fade + slide

Confidence updates

Crossfade only

Avoid exaggerated "robotic" effects.

---

# 24. Voice Interaction Motion

Microphone

Pulse while listening.

Waveform

Live amplitude visualization.

Transcription

Progressively revealed.

---

# 25. Maps

Map interactions use platform-native gestures.

Custom animations should be limited to:

* marker appearance
* route drawing
* technician movement

Camera movement should preserve orientation whenever possible.

---

# 26. Charts

Charts animate only on first appearance.

Avoid repeated animation during live updates.

Updates should interpolate values smoothly.

---

# 27. Empty States

Illustrations may contain subtle looping motion.

Requirements:

* low amplitude
* slow timing
* non-essential

Animation should stop when off-screen.

---

# 28. Gesture Physics

Dragging

Follows finger exactly.

Release

Uses spring settling.

Overscroll

Elastic resistance.

Momentum

Platform-native.

Never create conflicting gesture systems.

---

# 29. Reduced Motion

When Reduced Motion is enabled:

Replace:

* slide

with:

* fade

Replace:

* spring

with:

* opacity

Disable:

* parallax
* decorative looping animations
* large scaling effects

Essential feedback remains visible.

---

# 30. Performance Targets

Target frame rate

```text
60 FPS
```

Acceptable minimum

```text
55 FPS
```

Animation work must remain on the native/UI thread whenever possible.

Avoid layout recalculation during animation.

---

# 31. Animation Ownership

Animations belong to components.

Business logic must never directly manipulate animation state.

State changes trigger predefined motion behaviors.

---

# 32. Motion Hierarchy

Primary Motion

Navigation

Secondary Motion

Component state

Tertiary Motion

Decorative

Only one primary motion event should occur at a time.

---

# 33. Timing Coordination

Multiple animations should be coordinated.

Preferred pattern

```text
Container

↓

Children

↓

Details
```

Avoid simultaneous unrelated movement.

---

# 34. Accessibility Considerations

Motion must never:

* trigger seizures
* induce motion sickness
* hide important information
* require animation for comprehension

Animations should remain interruptible whenever possible.

---

# 35. Implementation Guidelines

Recommended React Native technologies:

* Reanimated
* Gesture Handler
* Native platform transitions
* Expo Router transition APIs

Avoid JavaScript-thread animations for frequently used interactions.

---

# 36. Motion Token Usage

All motion values must reference design tokens.

Example

```typescript
duration.motion.standard

easing.motion.enter

spring.default
```

Hardcoded timing values are prohibited.

---

# 37. Motion Testing

Every reusable animation should be verified for:

* duration
* easing
* interruption
* accessibility
* reduced motion
* performance

Visual regression testing should include transition states where feasible.

---

# 38. Anti-Patterns

The following are prohibited:

❌ Gratuitous animation

❌ Long splash animations

❌ Infinite decorative loops

❌ Delayed button feedback

❌ Flashing effects

❌ Multiple competing animations

❌ Non-deterministic transitions

❌ Animations that block user interaction

---

# 39. Future Motion Extensions

The motion architecture supports future enhancements including:

* Shared element navigation across platforms
* Predictive back navigation
* AI-assisted contextual transitions
* Adaptive motion based on device performance
* White-label motion profiles

These extensions must preserve the core motion principles and token architecture.

---

# 40. Relationship to the Design System

This specification integrates with the rest of the TaskPRO Design System:

* **TASKPRO-DS-001** provides motion tokens (durations, easing, opacity).
* **TASKPRO-DS-002** defines which components animate and their states.
* **TASKPRO-DS-003** defines when interactions trigger motion.
* **TASKPRO-DS-004** defines how those transitions behave.
* **TASKPRO-DS-005** will define how motion adapts across light, dark, accessibility, and future branded themes.
