# AUTH-001 — Authentication, Authorization & Security Architecture v0.1

## Purpose

Authentication is not just about signing users in—it establishes **identity**, **trust**, and **permissions** across the entire TaskPRO platform.

The architecture should be:

* Secure by default
* Stateless
* Role-based
* Permission-driven
* Multi-device
* Offline-aware
* Auditable
* Extensible

Authentication should be completely independent from business modules.

---

# Authentication Principles

TaskPRO should distinguish five different concepts:

```text
Authentication
    Who are you?

↓

Identity
    Who is this person?

↓

Authorization
    What can they do?

↓

Verification
    Can they be trusted?

↓

Session
    Are they currently authenticated?
```

Keeping these separate prevents security logic from leaking into the domain.

---

# Identity Model

Every authenticated person is a **User**.

A User may own one or more Personas.

```text
User

├── Customer
├── Professional
├── Administrator
├── Support Agent
└── Future Personas
```

This is preferable to a single `role` field because one person can legitimately operate in multiple capacities without maintaining separate accounts.

Example:

* A homeowner who also provides electrical services.
* An administrator who occasionally books services.
* A franchise owner who is also a technician.

# Authorization Model

Avoid hard-coded role checks such as:

```typescript
if (user.role === "admin")
```

Instead, use permissions.

```text
Permission

booking:create

booking:update

booking:cancel

booking:assign

payment:refund

review:delete
```

Roles become collections of permissions.

```text

↓

↓

Permissions

↓

↓

```

# JWT Claims

The JWT should remain minimal.

```text
sub

session_id

issued_at

expires_at
```

Avoid embedding business data such as ratings, permissions, or profile information inside tokens.

These values change frequently and belong in the database.

---

# Row-Level Security (RLS)

Supabase RLS should be the first security boundary.

Examples:

Customer:

```text
Can only read

Own profile

Own bookings

Own messages

Own invoices
```

Professional:

```text
Can read

Assigned bookings

Own portfolio

Own availability

Own payments
```

Administrator:

```text
Can access according to granted permissions.
```

No API should rely solely on client-side filtering.

# Offline Authentication

The mobile app should support limited offline functionality.

Local storage:

* Encrypted session token (Expo SecureStore)
* User profile cache
* Permission cache
* Feature flags

Offline users can:

* View cached data
* Complete permitted offline workflows
* Queue commands

They cannot perform operations requiring server validation until connectivity is restored.

# Architectural Observation

I recommend **moving beyond traditional RBAC (Role-Based Access Control) to a hybrid RBAC + PBAC (Policy-Based Access Control) model**.

Roles answer **who generally has access**, while policies answer **whether this specific action is allowed in this context**.

For example:

* A professional can only update **their own** availability.
* An administrator can refund payments **up to a configured limit**.
* A support agent can view a booking **only while assigned to the case**.

This combination provides significantly greater flexibility while keeping the permission model understandable and maintainable.
