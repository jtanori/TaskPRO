# DB-001 — Canonical Database Schema v0.1

## Purpose

The database is **not** an implementation detail—it is the canonical persistence model for the TaskPRO domain.

The schema should be:

* Domain-first
* Event-aware
* Audit-friendly
* Multi-tenant ready
* Offline synchronization capable
* AI-ready
* Realtime by design

The database should never mirror the UI. Instead, the UI projects data from the domain.

---

# Technology

**Database**

* PostgreSQL 17 (Supabase)
* UUIDv7 primary keys
* Row Level Security (RLS)
* Realtime enabled
* JSONB for extensibility
* Generated TypeScript types
* pgvector for AI features
* PostGIS (optional, if Supabase project supports it) for geospatial queries

# Administration Schema


# Soft Deletes

Instead of physical deletes, all major tables should implement:

```
deleted_at
deleted_by
```

This preserves referential integrity and supports compliance and recovery.

---

# Common Columns

Every business table should include:

```
id
created_at
updated_at
created_by
updated_by
deleted_at
deleted_by
version
```

The `version` column enables optimistic concurrency control, preventing conflicting updates in collaborative or offline scenarios.

---

# Referential Principles

* Foreign keys enforce consistency between aggregates.
* Lookup tables (e.g., statuses, categories) should be immutable where possible.
* Junction tables model many-to-many relationships explicitly.
* Business identifiers (e.g., `booking_number`) are separate from UUID primary keys.

# Architectural Observation

One enhancement I recommend beyond the initial proposal is introducing **schema-level ownership**. Each bounded context owns its own PostgreSQL schema, migrations, repositories, and domain services. This mirrors the module organization in the Expo application and makes it straightforward to evolve the platform toward independently deployable services if that ever becomes desirable, while remaining a single Postgres database today.

This approach also aligns well with the event-driven architecture defined in `DOM-001`, giving each bounded context clear ownership of its persistence model without sacrificing transactional consistency.
